## Vision: Agentic Many-Body Physics

This is NOT a task-delegation framework ("agents work for me"). This is an **interaction framework** where agents are first-class independent entities on equal footing.

### Metaphor
Most multi-agent frameworks are **mean-field theory**: central orchestrator = external potential, agents move in that field.

This system is **N-body quantum mechanics**:
- Each agent has its own state (wavefunction)
- Coupling is LOCAL (agents interact with neighbors, not global broadcast)
- Topology defines who sees whom (not "everyone in a group chat")
- Behavior is EMERGENT from local rules, not orchestrated

### Architectural Principles

| Mean-Field (CrewAI/AutoGen) | Many-Body (Our Target) |
|----------------------------|------------------------|
| Central orchestrator/supervisor | Decentralized — no central controller |
| Global message bus (all see all) | Local topology — who-sees-whom is configurable |
| User initiates every interaction | Agents can initiate interactions asynchronously |
| Static agent roles | Dynamic agent state that evolves |
| Hierarchical (manager → workers) | Flat — all agents equal |

## Core Concepts

### 1. Agent = Independent Process
Each agent is NOT a function call. It is a persistent entity with:
- Its own `ChatEngine` + `LLMAdapter` + optional `ToolExecutor`
- Its own message inbox (not a shared thread)
- Its own internal state / memory
- Ability to send messages to other agents WITHOUT user prompting

### 2. Topology (Who Sees Whom)
Instead of "all agents in a room", define coupling:
```typescript
interface Topology {
  // Graph: nodes are agents, edges are visibility
  neighbors(agentId: string): string[];
  // Or: agent A can see messages from agent B?
  canReceiveFrom(receiver: string, sender: string): boolean;
}
```

Examples:
- **Fully connected**: complete graph (current group chat)
- **Ring**: each agent sees only 2 neighbors
- **Star**: hub agent sees all, leaf agents see only hub
- **Custom**: physics-inspired (nearest-neighbor on lattice, small-world, etc.)

### 3. Message Bus (Not Thread-Based)
Current chat: all messages in one array, everyone sees everything.
Many-body: each agent has its own inbox. Messages are routed by topology.

```typescript
interface AgentInbox {
  agentId: string;
  messages: AgentMessage[];  // only messages this agent can see
  unread: boolean;
}
```

### 4. Emergent Behavior
Agents can:
- React to messages from neighbors (response)
- Initiate conversations without user prompt (spontaneous)
- Form coalitions / clusters dynamically
- Exhibit phase transitions (sudden consensus, polarization, etc.)

## Implementation Plan

### Phase A: Basic Orchestrator (MVP)
Build the existing T6 spec but with topology awareness:
- `Orchestrator` class with `Topology` injection
- `dispatch()` respects topology (not full broadcast)
- Sequential / parallel / debate modes
- `@AgentName` mention routing (T7)

### Phase B: Independent Agent Lifecycle
- Agents run in their own async loops
- Can receive messages while user is idle
- `Agent.processInbox()` — agent reads inbox, decides to respond
- `Agent.sendMessage(toAgentId, content)` — agent-initiated communication

### Phase C: Emergent Dynamics
- Agents can spawn child agents
- Agents can terminate (self-destruct)
- State evolution over time (memory, preferences, "mood")
- Observable collective phenomena (consensus, oscillation, chaos)

## API Sketch

```typescript
interface Topology {
  neighbors(agentId: string): string[];
  addEdge(a: string, b: string): void;
  removeEdge(a: string, b: string): void;
}

class RingTopology implements Topology { /* each sees 2 neighbors */ }
class FullyConnectedTopology implements Topology { /* complete graph */ }
class StarTopology implements Topology { /* hub + leaves */ }

interface AgentEngine {
  id: string;
  name: string;
  color: string;
  adapter: LLMAdapter;
  toolExecutor?: ToolExecutor;
  inbox: AgentInbox;
  processInbox(): Promise<void>; // agent decides what to do
}

class ManyBodyOrchestrator {
  constructor(options: {
    engines: AgentEngine[];
    topology: Topology;
    mode: 'sequential' | 'parallel' | 'debate' | 'free';
  });
  
  // User sends message → routed by topology
  userMessage(text: string): AsyncGenerator<AgentResponse>;
  
  // Agents run their own loops
  startAgentLoops(): void;
  stopAgentLoops(): void;
  
  // Observe emergent state
  getCollectiveState(): CollectiveState;
}
```

## Design Decisions
- **No central controller** — topology defines flow, not a manager
- **Error isolation** — if one agent fails, topology reroutes
- **User is just another node** — user is an agent in the topology, not external god
- **Observability first** — must be able to visualize topology + message flow
