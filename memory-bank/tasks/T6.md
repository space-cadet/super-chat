## Details

Enable multiple agents to participate in a single conversation. The user types once, and one or more agents respond — either sequentially or in parallel.

## Architecture

```
Orchestrator
├── engines: Map<agentId, AgentEngine>
├── mode: 'sequential' | 'parallel'
├── contextStrategy: 'full' | 'isolated'
│
├── parseAndRoute(text): { targets, cleanText }
│   └── Extract @AgentName mentions
│
├── dispatch(text, thread): AsyncGenerator<AgentResponse>
│   ├── Sequential: for each agent → run → yield response
│   └── Parallel: Promise.all → yield as they complete
│
├── buildContext(agentId, thread, cleanText): Message[]
│   ├── Full: agent sees all messages with attribution
│   └── Isolated: agent sees only user + its own responses
│
└── debate(text, thread): AsyncGenerator<AgentResponse>
    ├── Round 1: all agents respond to user
    └── Round 2: agents see each other's responses, can add follow-ups
```

## Key Features
- Mention-based routing: `@Cloudy fetch arxiv` → only Cloudy acts
- Sequential mode (default): agents build on each other's responses
- Parallel mode: all agents respond simultaneously
- Full transparency: agents see complete conversation
- Isolated mode: agents only see their own context
- Debate mode: agents can reply to each other

## API

```typescript
interface AgentEngine {
  id: string;
  name: string;
  color: string;
  adapter: LLMAdapter;
  toolExecutor?: ToolExecutor;
}

interface AgentResponse {
  agentId: string;
  agentName: string;
  message: ChatMessage;
  tokenEstimate?: number;
}

class Orchestrator {
  constructor(options: OrchestratorOptions);
  
  addEngine(engine: AgentEngine): void;
  removeEngine(agentId: string): void;
  
  parseAndRoute(text: string): { targets: AgentEngine[], cleanText: string };
  dispatch(text: string, thread: ChatMessage[]): AsyncGenerator<AgentResponse>;
  debate(text: string, thread: ChatMessage[]): AsyncGenerator<AgentResponse>;
}
```

## Design Decisions
- Each agent has its own LLM adapter and optional tool executor
- Context strategy configurable per orchestrator instance
- Error isolation — if one agent fails, others continue
- Agent identity preserved in messages (agentId, agentName, agentColor)
