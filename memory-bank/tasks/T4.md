## Details

Generic tool execution wrapper that bridges ToolAdapter and AgentLoop. Supports dynamic tool registration, batch execution, and error recovery.

## Architecture

```
ToolExecutor
├── adapter?: ToolAdapter (fallback for static tools)
├── handlers: Map<string, ToolHandler> (dynamic registration)
│
├── getAvailableTools() → ToolDefinition[]
│   └── Delegates to adapter if present
│
├── execute(call: ToolCall) → ToolResult
│   ├── Check handlers map first (dynamic tools)
│   ├── Fall back to adapter.executeTool() (static tools)
│   └── Catch errors → ToolResult with error field
│
└── executeBatch(calls: ToolCall[]) → ToolResult[]
    └── Promise.all(calls.map(execute))
```

## Design Decisions

1. **Two-tier execution** — Dynamic handlers (runtime registration) checked before adapter fallback
2. **Error wrapping** — All exceptions caught and returned as `ToolResult` with `error` field
3. **Result serialization** — Non-string results JSON-stringified for LLM consumption
4. **Framework-agnostic** — No Obsidian APIs; pure TypeScript

## Comparison with obsidian-ai

| Aspect | obsidian-ai ToolExecutor | super-chat ToolExecutor |
|--------|-------------------------|------------------------|
| Coupling | Tightly coupled to Obsidian (`App`, `TFile`, `vault`) | Framework-agnostic |
| Tools | Hardcoded 13 Obsidian tools | Delegates to `ToolAdapter` |
| Registration | None (fixed tool set) | Dynamic via `register()` |
| Formatting | Tool-specific `formatToolResult()` in AgentLoop | Pluggable `ToolResultFormatter` |

## API

```typescript
class ToolExecutor {
  constructor(opts?: { adapter?: ToolAdapter })

  register<T>(name: string, handler: ToolHandler<T>): void
  getAvailableTools(): ToolDefinition[]
  execute(call: ToolCall): Promise<ToolResult>
  executeBatch(calls: ToolCall[]): Promise<ToolResult[]>
}
```
